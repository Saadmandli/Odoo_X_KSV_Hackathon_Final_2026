# Deploying EcoMiles

Three free tiers, no credit card. Roughly 25 minutes end to end.

| Piece | Host | Why this one |
|---|---|---|
| Postgres | **Neon** | Free forever. Render's free Postgres expires 30 days after creation |
| API | **Render** | A long-lived Node process, so no serverless cold start on every request |
| Frontend | **Vercel** | Static build, instant, never sleeps |

You need accounts on all three. Each signs in with GitHub, so there is nothing
to fill in beyond authorising the app.

---

## 1 — Database (Neon)

1. Create a project at [neon.tech](https://neon.tech).
2. Copy the **pooled** connection string — the host contains `-pooler`.
   Prisma opens many connections and the direct string exhausts them.
3. Seed it from your laptop. Render's free tier has no shell, so this is the
   only chance to load the demo organisation:

   ```bash
   cd server
   CARPOOL_DATABASE_URL="<neon-pooled-url>" npx prisma migrate deploy
   CARPOOL_DATABASE_URL="<neon-pooled-url>" npm run db:seed
   ```

   On PowerShell, set it first instead:

   ```powershell
   $env:CARPOOL_DATABASE_URL="<neon-pooled-url>"
   npx prisma migrate deploy
   npm run db:seed
   ```

## 2 — API (Render)

1. **New → Blueprint**, point it at this repository. `render.yaml` provisions
   the service, so there is nothing to configure by hand.
2. In the dashboard set the two variables it cannot know:
   - `CARPOOL_DATABASE_URL` — the Neon pooled string
   - `CORS_ORIGINS` — the Vercel URL, added once step 3 gives you one.
     Exact origin, **no trailing slash**.
3. Wait for the first deploy, then check `https://<api>.onrender.com/api/health`
   returns `{"ok":true}`.

`JWT_SECRET` is generated by Render. **Do not rotate it mid-demo** — every
signed-in session is invalidated the moment you do.

## 3 — Frontend (Vercel)

1. Import the repository. Set **Root Directory** to `client`.
2. Add the environment variable **before the first build**:

   ```
   VITE_API_URL = https://<api>.onrender.com/api
   ```

   Vite inlines `VITE_` values at build time. A bundle built without this
   silently calls `localhost` and every request fails — and rebuilding is the
   only fix, so set it first.
3. Deploy, then go back to Render and put the resulting URL in `CORS_ORIGINS`.

`client/vercel.json` rewrites every path to `index.html`. Without it a direct
link to `/trips` or a shared `/track/<token>` returns a 404, because those
routes exist only in the browser.

## 4 — Keep the API awake

Free Render services sleep after 15 minutes idle and take ~50 seconds to wake.
Point [cron-job.org](https://cron-job.org) or UptimeRobot at
`https://<api>.onrender.com/api/health` every 10 minutes.

744 hrs/month sits under the 750-hour free cap — but only for **one** service,
which is why the frontend is on Vercel rather than a second Render service.

## Before you demo

- **Re-seed if you have run the test scripts.** `scripts/smoke*.mjs` and
  `audit.mjs` publish real rides and bookings; they leave the history and the
  reports charts full of near-identical junk. `npm run db:seed` restores the
  intended data.
- **Wake the API** a few minutes beforehand by loading the health URL.
- **Rehearse the live-tracking steps.** Driver Start trip → marker moves →
  share the link → open it in a tab with no login. That sequence is the one
  worth being boring about.

## If something breaks

| Symptom | Cause |
|---|---|
| Every request fails on the deployed site | `VITE_API_URL` was missing at build time. Set it and **redeploy** |
| Requests blocked by CORS | `CORS_ORIGINS` does not exactly match the Vercel origin, or has a trailing slash |
| `/track/<token>` 404s | `client/vercel.json` was not picked up — confirm Root Directory is `client` |
| Login works, then everything 401s | `JWT_SECRET` changed between deploys |
| First request after a quiet spell hangs | Render cold start. This is the ~50s wake, not a fault |
